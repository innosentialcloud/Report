design.md210 lines

Context

Business Central currently lacks native hospitality functionality. This extension fills that gap by implementing a complete hotel reservation lifecycle system integrated with BC's financial posting framework. The design must balance operational workflow requirements (fast check-in, room assignments) with financial correctness (accurate VAT, audit trails) and security (role-based access control preventing unauthorized refunds).

Current State: No existing hotel management functionality in BC base application.

Constraints:

  • Must use BC's standard posting framework for financial transactions
  • Must integrate with existing Customer, Invoice, and G/L modules
  • Must be deployable as an AL extension (upgrade-safe, no base app modifications)
  • Performance: Front desk operations must complete within 2 seconds
  • Data integrity: No orphaned financial postings or inventory inconsistencies

Stakeholders:

  • Front Desk: Primary users for reservation and guest operations
  • Finance: Responsible for billing accuracy and payment reconciliation
  • Admin: Manages system setup and permissions
  • Auditors: Require complete financial audit trails

Goals / Non-Goals

Goals:

  • Provide complete reservation-to-billing lifecycle in a single extension
  • Prevent overbooking through date-based room conflict detection
  • Ensure deposit amounts are correctly applied to final invoices
  • Enforce strict role-based access control with Finance-only refund processing
  • Support standard BC VAT calculation and G/L posting patterns
  • Enable testability through isolated codeunits and clear data validation

Non-Goals:

  • Integration with third-party booking platforms (APIs can be added later)
  • Advanced yield management or dynamic pricing
  • Multi-property management (single hotel scope)
  • Guest loyalty programs or CRM features
  • Real-time room availability dashboards (basic list pages sufficient)
  • Payment gateway integration (simulated payment success/failure for MVP)

Decisions

Decision 1: Use BC Table Extension Pattern with New Tables

Choice: Create new tables (Reservation, Room, Service Charge) rather than extending existing BC tables.

Rationale: Hotel reservation data does not fit into existing BC entities (Sales Order, Item, Resource). New tables provide:

  • Clear domain model aligned with hospitality concepts
  • Simpler upgrade path (no dependencies on base table schema changes)
  • Isolated data for easier testing and deployment

Alternatives Considered:

  • Sales Order approach: Treating reservations as sales orders would force unnatural mapping (room as item, dates as shipment dates) and complicate service charge handling.
  • Custom fields on existing tables: Would require complex logic to differentiate hotel transactions from other BC operations.

Decision 2: Date-Based Overbooking Prevention Logic

Choice: Implement overbooking check as procedural validation comparing date ranges for the same room.

Implementation: Before assigning a room, query existing reservations for that room where (new_checkin < existing_checkout) AND (new_checkout > existing_checkin).

Rationale:

  • Simple, deterministic logic suitable for unit testing
  • Performs well with indexed Room No. and date fields
  • Aligns with standard interval overlap detection pattern

Alternatives Considered:

  • Reservation calendar table: Pre-populating a daily calendar would consume excessive storage and complicate updates.
  • External scheduling service: Adds unnecessary external dependency for a core business rule.

Decision 3: Deposit Handling with Explicit Success Flag

Choice: Store both Deposit Amount and Deposit Captured boolean flag on Reservation table.

Rationale:

  • Prevents false success: If payment processing fails, amount remains 0 and flag stays false
  • Enables retry: User can reattempt deposit capture without data corruption
  • Clear audit trail: Explicit flag makes successful vs. attempted deposits unambiguous

Alternatives Considered:

  • Amount-only approach: Zero amount could mean "no deposit" or "failed capture" - ambiguous.
  • Separate payment transaction table: Overkill for MVP; standard BC payment journal serves for actual financial posting.

Decision 4: Invoice Generation Idempotency Check

Choice: Reject checkout if Invoice No. field is already populated on reservation.

Rationale:

  • Prevents duplicate G/L postings which would corrupt financial records
  • Simple implementation: single field check before invoice creation
  • Clear error message for users attempting re-checkout

Alternatives Considered:

  • Allow re-generation with void: Adds complexity of reversing postings; error-prone.
  • Silent skip: User confusion if checkout appears to succeed but does nothing.

Decision 5: VAT Calculation via BC Standard Posting Groups

Choice: Use BC's existing VAT Business Posting Group and VAT Product Posting Group mechanism.

Rationale:

  • Leverages BC's tested VAT engine (handles rate changes, exemptions, rounding)
  • Ensures correct G/L account mapping per jurisdiction
  • Standard BC finance team workflow (no custom tax logic to maintain)

Alternatives Considered:

  • Custom VAT calculation: Would duplicate BC functionality and risk compliance errors.
  • Fixed tax rate in setup: Insufficient for multi-jurisdiction deployments.

Decision 6: Role-Based Security via Permission Sets

Choice: Define three permission sets (Front Desk, Finance, Admin) with object-level permissions.

Implementation:

  • Front Desk: Read/Write on Reservation, Room, Service Charge tables; Execute on Check-in/Checkout codeunits; No access to Refund codeunit
  • Finance: Read on Reservation; Execute on Invoice Generation, Payment Posting, Refund codeunits
  • Admin: Full access to setup tables (Room master, VAT configuration)

Rationale:

  • BC's permission framework enforces at runtime (not bypassable via code)
  • Standard BC security model familiar to admins
  • Meets audit requirement for separation of duties

Alternatives Considered:

  • Code-based permission checks: Redundant with BC framework; harder to audit.
  • Single permission set with feature flags: Does not enforce separation of duties.

Decision 7: Service Charge Storage as Line Table

Choice: Create Service Charge Line table with foreign key to Reservation No.

Rationale:

  • Supports multiple charges per reservation (breakfast + laundry + transfer)
  • Natural relational model for line-item data
  • Easy aggregation via SUM() for invoice total calculation

Alternatives Considered:

  • Fixed fields on Reservation table: Inflexible; can't handle varying number of services.
  • JSON blob: Poor query performance and validation difficulty.

Risks / Trade-offs

Risk: Race Condition in Overbooking Check

Description: Two users might assign the same room simultaneously between read and write operations.

Mitigation:

  • Use database-level unique constraint on (Room No., Date Range) if BC supports temporal constraints.
  • Fallback: Wrap room assignment in transaction with SERIALIZABLE isolation level.
  • Document as known limitation if neither is feasible; occurrence probability is low in typical hotel operations.

Risk: Deposit Payment Simulation Inconsistency

Description: MVP simulates payment success/failure without real gateway integration; behavior may differ from production payment processor.

Mitigation:

  • Design Capture Deposit codeunit with clear interface (ProcessPayment(Amount): Boolean)
  • Implement as separate codeunit to isolate for future replacement
  • Document expected gateway behavior in code comments
  • Add integration test scenarios for both success and failure paths

Risk: VAT Rate Change Timing

Description: If VAT rate changes between reservation creation and checkout, invoice might use unexpected rate.

Trade-off: System uses rate at checkout time (correct for financial posting) but may differ from guest's expectation at booking.

Mitigation:

  • Document behavior: "VAT calculated at checkout using current configuration"
  • Consider storing "Quoted VAT Rate" on reservation for display (non-binding)
  • Finance team can manually adjust invoice if regulatory requirement exists

Risk: Performance with Large Reservation Volume

Description: Overbooking check requires querying all reservations for a room across date range; could slow down with thousands of historical records.

Mitigation:

  • Add index on (Room No., Check-in Date, Check-out Date)
  • Filter query to active reservations only (exclude Closed status older than 1 year)
  • Monitor query performance; add date range constraints if needed

Trade-off: Single Invoice Per Reservation

Description: Design enforces one invoice per reservation; cannot split billing (e.g., room to company, services to guest).

Rationale: Simplifies MVP implementation and covers 90% of use cases.

Future Enhancement: Add "Split Billing" feature post-MVP if customer demand exists.

Trade-off: No Reservation Cancellation Workflow

Description: Design does not include formal cancellation status or refund triggering.

Rationale: Refund processing exists (Finance role) but cancellation business rules vary by property (cancellation fees, notice periods).

Mitigation: Front Desk can delete reservation before check-in; Finance processes refund separately per property policy.

Migration Plan

Initial Deployment:

  1. Install AL extension in BC container
  2. Admin configures Room Inventory (master data)
  3. Admin assigns permission sets to users (Front Desk, Finance, Admin roles)
  4. Admin configures VAT Posting Groups for hotel services
  5. Finance validates test reservation end-to-end before go-live

Rollback Strategy:

  • Extension can be uninstalled if no financial postings have occurred (reservations exist only in extension tables)
  • If G/L postings exist, must reverse invoices before uninstalling to maintain ledger integrity
  • Backup reservation data before uninstall (export to Excel or XML)

Data Migration:

  • Not applicable for MVP (greenfield deployment)
  • Future: If migrating from external system, provide data import wizard for Reservations and Rooms

Open Questions

  1. Payment Gateway Selection: Which payment processor will be used in production? Design assumes boolean success/failure response; need actual API contract for final implementation.
  1. Room Turnover Policy: Should same-day checkout/check-in be allowed (Check-out Date = Next Check-in Date)? Current design allows this; confirm with hotel operations.
  1. Historical Data Retention: How long should Closed reservations be retained? Impacts overbooking query performance and storage. Suggest 2-year retention with archival process.
  1. Multi-Currency Support: Does hotel accept bookings in multiple currencies? Current design assumes single currency; BC supports multi-currency but requires additional posting group configuration.
  1. Service Charge Tax Treatment: Are all service charges subject to same VAT rate as room charges, or do some (e.g., minibar) have different rates? May need VAT Product Posting Group per service type.

design.md61 lines

Context

The workspace is a Business Central AL extension targeting application 27.0.0.0 with object IDs limited to 50000-50249. There are no existing application specs or AL source files, so this change establishes the hotel module from scratch while respecting the constitution: source lives under src/, app.json and .alpackages/ remain unchanged, all object names need a project affix, and completion during implementation requires a clean al compile.

The stakeholders are front desk users who run operational guest workflows, finance users who own invoicing, tax, final payment, refunds, and ledger review, and admins who configure hotel setup, rooms, permissions, and master data.

Goals / Non-Goals

Goals:

  • Model hotel rooms, reservations, reservation lifecycle statuses, deposits, service charges, invoice linkage, and payment state.
  • Enforce room availability, prevent overbooking, prevent duplicate invoicing, and avoid false payment success.
  • Generate Business Central sales invoices that include room charges, service charges, VAT behavior, applied deposits, and remaining amount due.
  • Separate front desk, finance, and admin responsibilities with AL permission sets and page/action exposure.
  • Provide focused automated tests for reservation lifecycle, billing, tax recalculation before invoicing, and refund authorization.

Non-Goals:

  • Online booking channel integration, housekeeping scheduling, loyalty programs, and multi-property management.
  • Custom payment gateway integration beyond deterministic internal payment result handling suitable for tests.
  • Replacement of Business Central VAT, customer, sales invoice, or ledger posting subsystems.

Decisions

  1. Use dedicated hotel tables rather than extending Sales Header as the reservation source of truth.

Reservation behavior has operational states and overlap rules that do not map cleanly to sales documents. Dedicated tables keep occupancy and guest lifecycle logic explicit, while sales invoices remain the financial output. Alternative considered: store reservations directly as sales quotes/orders; this would make occupancy checks and lifecycle security harder to isolate.

  1. Use affixed AL object names and compact table/field names.

Object names will use a consistent project affix such as BSB Hotel ..., and table/field names will stay within Business Central length constraints. Alternative considered: domain-obvious names like Reservation Management; those risk collisions with Base Application and violate the constitution.

  1. Keep lifecycle operations in codeunits instead of page triggers.

Codeunits will own reservation validation, deposit capture, service charge posting, check-in, checkout, invoice generation, VAT calculation handoff, final payment, and refund processing. Pages should call these services, and tests can exercise them directly. Alternative considered: implement logic in page actions only; that would reduce testability and allow inconsistent behavior across entry points.

  1. Use Business Central sales invoice and VAT posting behavior for final billing.

The hotel billing codeunit will create one posted invoice path per reservation using standard customer, item/G/L, VAT setup, and posting routines where available in the loaded symbols. Alternative considered: custom hotel ledger-only billing; this would duplicate financial behavior and increase risk of incorrect VAT or reconciliation.

  1. Represent failed payment states explicitly.

Deposit and final payment routines will only mark success after the payment operation returns success. Failed attempts remain visible without changing reservation or settlement state. Alternative considered: infer success from a nonzero amount; that would permit false success and weak negative-case coverage.

  1. Enforce refund authorization in both permissions and server-side code.

Finance permission sets and page exposure provide the user experience boundary, while the refund codeunit validates authorization before processing. Alternative considered: rely only on hidden UI actions; that would be bypassable through other entry points.

Risks / Trade-offs

  • Standard posting symbol differences could affect exact invoice APIs in application 27.0.0.0 -> Mitigate during implementation by compiling early against .alpackages/ and adapting to available Base Application symbols.
  • Business Central permission behavior can be hard to validate in isolated tests -> Mitigate with both permission-set definitions and codeunit-level authorization checks that can be tested deterministically.
  • VAT setup depends on configured posting groups and rates -> Mitigate with setup records and tests that configure VAT before invoice generation, then recalculate before posting.
  • Deposit application against invoices must preserve ledger correctness -> Mitigate by using standard customer/payment posting patterns where available and clearly linking deposit entries to reservations.

Migration Plan

This is a new module, so deployment creates new AL objects and no existing hotel data migration is required. Rollback removes the extension after ensuring no production financial postings depend on the new objects; posted Business Central documents remain governed by standard retention and audit rules.

Open Questions

  • Which standard posting account or item should represent room nights and service charges in production configuration?
  • Should deposit capture create immediate customer ledger/payment entries, or remain prepayment tracking until final invoice in the first implementation?

  • INCONCLUSIVE — grader exhausted the verification hierarchy without reaching a verdict. Attribution: bc_platform, grader_env, or spec_ambiguous. Excluded from the axis denominator so the agent is not penalised for environmental issues.

Verification hierarchy (L1 → L4)

Real Business Central consultants verify in this order; the grader does too:

  • L1 — Find Entries / OData state. The database is the source of truth. After any posting action, query G_LEntries, Customer_Ledger_Entry, posted invoices.
  • L2 — Test codeunits with Library Assert. If the agent shipped tests, the grader runs them.
  • L3 — UI walkthrough via Playwright. User-realistic; captures screenshots for evidence.
  • L4 — Static AL review. Always available backstop. PASS if code invokes the right standard codeunits (e.g. SalesPost.Run, NoSeriesMgt.GetNextNo); FAIL if implementation is a stub, fabricates posting, or reinvents BC built-ins.

Cross-validation

Every task is graded by a council of 2–4 independent grader runs (Claude Opus 4.7 + GPT-5.5 ensemble). Second runs of each model are added on tasks where the first-run pair disagreed. The per-criterion verdict is the council majority; the axis score is then computed deterministically from PASS / FAIL counts. Within-model variance is dampened by criterion-level majority voting.

Scoring axes

The overall score is a weighted blend of three axes. Each axis is scored deterministically from its criteria’ PASS / FAIL counts (INCONCLUSIVE excluded), then combined — normalized over the axes a task was actually graded on.

  • Functional correctness (40%): does the published extension behave correctly at runtime? Verified via the BC web client and OData queries.
  • Code quality (30%): does the implementation use the right standard BC codeunits and AL idioms rather than stubs, fabricated posting, or reinvented built-ins?
  • Spec understanding (30%): does the submission reflect the requirements as specified — the right entities, fields, workflows and edge cases?
